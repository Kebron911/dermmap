Security & compliance

HIPAA compliance you can actually explain to your compliance officer.

DermMap's security architecture was designed from the first line of code as a clinical application — not adapted from a consumer product after the fact.

Every access control, encryption layer, and audit trail in DermMap exists because of a specific HIPAA requirement or clinical data security principle. Here's how it maps.

Full HIPAA Compliance Implementation

Requirement DermMap Implementation
Encryption at rest AES-256 for all stored data — patient records, lesion data, photos, audit logs
Encryption in transit TLS 1.3 (minimum TLS 1.2) for all API calls and data transfers
Unique user identification Individual accounts required — shared logins are blocked
Automatic logoff 5-minute timeout on mobile, 15-minute timeout on web; PIN/biometric resume
Audit controls Immutable audit log for every action — user, timestamp, action type, affected record
Audit retention 6 years minimum, per HIPAA requirements
Access control (role-based) MA, Provider, Admin — minimum necessary access principle enforced
Emergency access Break-glass access with mandatory justification logging
PHI on device Photos stored only in encrypted app sandbox — never in camera roll
EXIF data Stripped from all photos before storage (EXIF embeds GPS coordinates — PHI risk)
Screen protection Screenshot and screen recording disabled within the app (iOS/Android)
Remote wipe Enrolled device remote wipe capability for lost/stolen devices
BAA Business Associate Agreement available with HIPAA-eligible cloud infrastructure
Infrastructure Dedicated VPC, no public-facing databases, encrypted backups, point-in-time recovery
